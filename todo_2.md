## Спецификация учебного DI-контейнера (Java + Clojure-конфигурации + scoped-proxy)

# 0. Цель проекта и рамки (scope)

### 0.1 Цель
Реализовать DI-контейнер для Java (в духе Spring Core), поддерживающий:
- внешние метаданные (конфиг) управляемых классов
- Java API получения объектов и мониторинга
- стандартные `javax.inject` аннотации: `@Inject`, `@Named`, `Provider<T>`
- поддержка `singleton`, `prototype`, `thread` scope
- scoped proxy: если singleton зависит от thread-scoped, внедрять прокси

### 0.2 Рамки (что не требуется)
Не реализуем Spring-подобные фичи вне ТЗ (по умолчанию):
- component scan
- `@Component`, `@Service`, `@Configuration`, `@Bean`
- AOP/транзакции/перехватчики
- Spring lifecycle (`@PostConstruct`, `@PreDestroy`, `BeanPostProcessor`, etc.) — только если позже явно добавите
- автоматическое “собери все бины в `List<T>` по типу” (вы решили: коллекции только явно в конфиге)

# 1. Конфигурации (Clojure) и внешние метаданные

## 1.1 Формат
- Конфигурации пишем на Clojure (рекомендуемый стиль — data-only, EDN-подобные структуры: map/vector).
- Конфиг описывает список (или map) определений бинов (`BeanDefinition`).

## 1.2 Обязательные поля метаданных (по ТЗ)
Каждый бин в конфиге должен иметь метаданные:

1) **класс реализации** (его имя)  
2) **имя** (обязательно, если есть несколько различных конфигураций для одного базового интерфейса)  
3) **тип жизненного цикла**: `singleton`, `prototype`, `thread`  
4) **параметры инициализации для set-методов** (setter injection)  
5) **параметры конструктора**

## 1.3 Имена и @Named — «истина» в конфиге
- **Единственный источник истины для имени/идентификатора бина**: `:name` в конфиге.
- `@Named` в Java-коде используется только как qualifier (уточнение выбора) при резолве по типу:
  - выбирает бин с соответствующим `:name` из конфига, когда кандидатов несколько.

## 1.4 Типы параметров, которые нужно поддержать
В конфиге должны поддерживаться:

### 1.4.1 Ссылки на другие бины
- Ссылки на другие определения бинов.

### 1.4.2 Примитивы и строки
- Примитивные типы (int, boolean и т.п.), boxed-типы, строки — **ДА**.

### 1.4.3 Коллекции
- Коллекции (например, `List`, `Map`) — **ДА**.
- Это “обычные классы/интерфейсы”, просто значения и вложенные структуры.
- **Массивы** (обычные `T[]`) — на старте **не трогаем**.

### 1.4.4 Коллекции бинов
- Коллекции поддерживаются **только как “явно перечисленные элементы в конфиге”**.
- Не реализуем Spring-фичу “дай мне `List<SomeInterface>` из всех кандидатов контейнера”.

## 1.5 Различение «литерал» vs «ссылка на бин»
Нужно соглашение, чтобы строка не была двусмысленной.

Принятое соглашение:
- `{:ref "beanName"}` — ссылка на бин по имени (из `:name` в конфиге)
- `{:value "abc"}` — литеральное значение

Это “по духу как Spring” (в XML тоже разделяют ref/value), но формат ваш.

# 2. Java API контейнера

## 2.1 Получение объектов
Контейнер предоставляет Java API для получения бинов:
- получение по типу
- получение по типу + имя (если нужно)
- (опционально) получение по имени, если решите добавить

Резолв по умолчанию:
- по типу
- при неоднозначности — уточнение `@Named` (или явный запрос с именем через API)

## 2.2 Мониторинг (минимальный must-have)
Минимальный набор мониторинга, который должен быть реализован:

1) `listBeanDefinitions()`  
   - список зарегистрированных бинов (имена, типы, scope, класс реализации)
2) `isInstantiatedSingleton(name/type)`  
   - создан ли уже данный singleton
3) `getDependencyGraph()`  
   - граф зависимостей “кто от кого зависит”
   - можно сначала сделать текстовый вывод; визуализацию/гистограммы — позже как расширение
4) счётчики обращений `getBean`
Дополнительно (как идеи/расширения, не must-have):
- время создания
- “dump thread-scope” (содержимое текущего thread local map)

# 3. Аннотации `javax.inject`: @Inject, @Named, Provider

## 3.1 Поддерживаемые аннотации
Обязательная поддержка:
- `@Inject`
- `@Named`
- `javax.inject.Provider<T>`

(Важно: по ТЗ именно `javax`, не `jakarta`)

## 3.2 Где обязаны работать `@Inject`
`@Inject` должен работать везде:
- на конструкторе (constructor injection)
- на полях (field injection)
- на методах/сеттерах (method/setter injection)

Если и поле, и сеттер помечены `@Inject` — **внедряем оба** (как в Spring).

## 3.3 Выбор конструктора (как в Spring)
- Если в классе есть **ровно один** конструктор, и `@Inject` нигде не стоит — используем этот конструктор.
- Если `@Inject` стоит:
  - должен быть **ровно один** “инъекционный” конструктор
  - если их несколько — ошибка с понятным сообщением

## 3.4 `@Named`
- `@Named` может встречаться “везде” (на полях, параметрах, методах, возможно на классах).
- Контейнер использует `@Named` при резолве:
  - если по типу несколько кандидатов — выбираем по имени (`:name` из конфига)
  - если кандидатов 0 → ошибка “не найден бин”
  - если кандидатов 2+ и `@Named` не помог → ошибка “неоднозначный выбор”

## 3.5 Provider<T>
### 3.5.1 Минимальная поддержка
- Поддерживаем `Provider<T>` **только как инъекцию зависимости** (точка внедрения).
- Не требуется (на старте) поддерживать получение `Provider<T>` из API контейнера.

### 3.5.2 Семантика
- `provider.get()` должен возвращать экземпляр `T` в момент вызова:
  - для `prototype`: **каждый `get()` возвращает новый экземпляр**
  - для `singleton`: будет возвращаться тот же singleton
  - для `thread`: будет возвращаться thread-scoped экземпляр текущего потока

# 4. Резолв зависимостей и ключи

## 4.1 Ключ резолва
- По умолчанию резолвим по **типу** (интерфейсу/классу).
- При конфликте (несколько кандидатов) используем `@Named`.
- Если конфликт не разрешён — ошибка `"ambiguous dependency"`.

## 4.2 “Inject через интерфейсы”, но классы тоже допускаются
Политика:
- Инъекция через интерфейсы — **рекомендуемый и “стабильный” путь** (и он нужен для scoped proxy через JDK proxy).
- Инъекцию по конкретному классу **разрешаем, но не гарантируем**.

Механика для “по классу”:
- ищем кандидатов, которые `assignableTo(requestedClass)` (т.е. реализация может быть подтипом requestedClass)
- 1 кандидат → ок
- 0 кандидатов → “not found”
- 2+ → “ambiguous”

Документация:
- явно пишем, что для корректной работы со scoped proxy ожидается, что зависимости описываются интерфейсами.

# 5. Создание бинов: constructor args + setter injection

## 5.1 Constructor args из конфига
- Аргументы задаются **по индексам**:
  - “как переданы — в том порядке и подставляем”
- Если в конфиге задано не все аргументы — **ошибка** (B5).

## 5.2 Setter injection из конфига
- Имена сеттеров задаются как **имя метода**, например `setCharacterName`.

## 5.3 Перегрузки (overload) и выбор “самого подходящего” метода
Если есть несколько методов `init(...)` / `setX(...)` с разными параметрами, то вызываем ошибку
- если найдено >1 подходящее перегруженное значение по имени сеттера → ошибка "ambiguous setter overload". То есть overload вообще не поддерживаем.
- при равной специфичности ошибка `"ambiguous setter"`

## 5.4 Переопределения (override) и “последний в дереве”
Пояснение вашей идеи:
- В цепочке наследования `Object -> Map -> HashMap -> MyHashMap`:
  - если метод переопределён в `HashMap` и не переопределён в `MyHashMap`, используется реализация `HashMap` (обычная Java-семантика).
- Практически: контейнер выбирает подходящую сигнатуру, а JVM выполняет dispatch корректно.

# 6. Жизненные циклы (scopes)

## 6.1 Поддерживаемые scope
- `singleton`
- `prototype`
- `thread`

## 6.2 Singleton
- Все singleton создаются **лениво** (по первому запросу).
- Thread-safe контейнер в целом не требуется, но:
  - при одновременном создании singleton в разных потоках необходимо обеспечить **at most once** (“чтобы dualtone случайно не получилось”).

Очистка singleton:
- не делаем explicit destroy; singleton живёт до конца программы.

## 6.3 Prototype
- новый экземпляр на каждый `getBean()`.
- prototype объекты не очищаем вручную: “оставляем на JVM (GC)”.

## 6.4 Thread scope
- один экземпляр на бин на поток (thread-local).
- хранение: нужно выбрать структуру хранения (см. ниже).
- thread-scope очищается вместе с потоком; явную очистку не реализуем в MVP.
* MVP: без очистки
* v1.1: добавить clearCurrentThreadScope()

### 6.4.1 Хранение thread-scoped объектов
Рекомендуемая модель:
- `ThreadLocal<Map<BeanId, Object>>`

Плюсы:
- простая реализация
- изоляция по потоку
- быстрый доступ

Минусы:
- в пуле потоков значения могут жить “дольше задачи”

### 6.4.2 BeanId
* Храним бины по уникальному String name из конфига (это главный идентификатор в контейнере).
* Делаем индекс для резолва по типу: `Map<Class<?>, List<BeanDefinition>>` — чтобы быстро находить кандидатов для get(type).
* Это удобно, потому что и @Named("..."), и {:ref "..."} напрямую ссылаются на name, а мониторинг тоже проще.
* Правило: `:name` обязателен и уникален во всём контейнере.

Для ThreadScope: `ThreadLocal<Map<String /*beanName*/, Object>>`


### 6.4.3 Очистка thread-scope
- Prototype: GC
- Singleton: живут до конца программы
- Thread scope:
  - thread-scope очищается вместе с потоком; явную очистку не реализуем в MVP”
  - СЕЙЧАС НЕ ДЕЛАЕМ: API типа `clearThreadScope()` / `clearCurrentThreadScope()` — **не обязательно**, но полезно

# 7. Scoped proxy (proxy-классы) и их назначение

**MVP-вариант: запрещаем injection thread-scoped в singleton и падаем с ошибкой “thread-scoped dependency into singleton requires proxy (not implemented yet)”, либо требуем Provider<T>.**

## 7.1 Когда прокси обязательны
Правило:
- Если singleton зависит от thread-scoped бина, контейнер **вместо реального объекта внедряет прокси**.

## 7.2 Что делает scoped proxy
Прокси:
- реализует тот же интерфейс зависимости
- при каждом вызове метода:
  1) резолвит актуальный экземпляр из контейнера для **текущего потока**
  2) делегирует вызов этому экземпляру

Это гарантирует корректную thread-scope семантику даже внутри singleton.

## 7.3 Технология прокси: JDK dynamic proxy
- Используем JDK dynamic proxy как самый простой способ в Java.
- Проксирование “для произвольных интерфейсов” не требует ручной генерации методов:
  - реализация методов делается через `InvocationHandler` и reflection (`Method.invoke`).
- Ограничение: JDK proxy работает только с интерфейсами:
  - поэтому документируем, что thread-scope proxy гарантирован для интерфейсов
  - инъекция по конкретным классам допускается, но без обещания проксирования

## 7.4 Требование “используйте Clojure для генерации proxy-классов”
Текущее решение:
- прокси генерируются средствами JVM (JDK dynamic proxy).
- Clojure используется как язык конфигурации.
- (Нужно явно проговорить с куратором, что JDK dynamic proxy считается допустимой “генерацией proxy-классов”; иначе придётся усложнять и реально генерировать классы через Clojure `gen-class` или подобные механизмы.)

# 8. Циклические зависимости

- Циклические зависимости контейнер **не исправляет**.
- Достаточно:
  - детектировать цикл при резолве/создании
  - упасть с понятным сообщением
  - желательно показывать цепочку резолва A → B → C → A

---

# 9. Ошибки и сообщения об ошибках

## 9.1 Тип исключений
- Можно бросать `RuntimeException` с понятным текстом (без отдельной иерархии исключений).

## 9.2 Требования к тексту ошибки
Сообщения должны быть подробными и полезными для пользователя фреймворка. Нужно покрыть минимум:

- “не найден бин”
- “найдено несколько бинов / неоднозначный выбор”
- “циклическая зависимость”
- “невозможно создать бин”:
  - нет подходящего конструктора
  - нет нужного сеттера
  - `IllegalAccess`/`InvocationTargetException` и т.п.
- Если один и тот же dependency задаётся и конфигом, и аннотациями — ошибка “double injection source”.

Дополнительно:
- включать **путь резолва** (цепочка A → B → C), особенно для циклов и not found.

# 10. Дополнительные договорённости по поведению "как в Spring"
- Конструктор по умолчанию: если он один — используем без `@Inject` (как в Spring).
- Если несколько конструкторов и нет `@Inject` —  ошибка.
- При неоднозначности по типу — ошибка, если не указан `@Named`.

# 11. Демо-приложение (mini-app)
Нужно сделать демонстрационный пример на 5–10 классов, который показывает:
- несколько реализаций интерфейса + выбор через `@Named`
- thread scope и работу scoped proxy (singleton → thread)
- prototype через `Provider<T>` (каждый get() новый)
- setter injection (включая перегрузки и выбор наиболее специфичного — чтоб показать ошибки)
- обработку ошибок (ambiguous / not found / cycle) — хотя бы 1–2 сценария
